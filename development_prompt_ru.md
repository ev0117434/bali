# ЗАДАЧА
Построить терминальное Python-приложение для полуавтоматической арбитражной торговли спот↔фьючерсы.

# ИСТОЧНИК СИГНАЛОВ
Приложение подписывается на Redis pub/sub канал `ch:spread_signals`.
Сигналы приходят в ДВУХ форматах — парсер должен сначала пробовать JSON, при ошибке — CSV:

## JSON формат:
```json
{
  "symbol": "THEUSDT",
  "direction": "A",
  "buy_exchange": "binance",
  "buy_market": "spot",
  "buy_ask": "0.2338",
  "buy_ask_qty": "5000",
  "sell_exchange": "gate",
  "sell_market": "futures",
  "sell_bid": "0.2362",
  "sell_bid_qty": "3000",
  "spread_pct": 1.0265,
  "ts_signal": 1773590375286
}
```

## CSV формат:
```
buy_exchange,sell_exchange.SYMBOL,buy_ask,sell_bid,spread_pct,ts_signal
binance,gate.THEUSDT,0.2338,0.2362,1.0265,1773590375286
```

# БИРЖИ
Поддерживаемые: binance, bybit, gate.io (gateio в ccxt), okx.
Использовать библиотеку ccxt. Для каждой биржи нужны ДВА инстанса: spot (defaultType: spot) и futures (defaultType: swap).
Символы фьючерсов в ccxt: BASE/QUOTE:QUOTE (например THE/USDT:USDT для USDT-маржинального линейного перпа).
Символы спота: BASE/QUOTE (например THE/USDT).
OKX требует apiKey + secret + password (passphrase).

# ЛОГИКА ЦЕН ВХОДА (КРИТИЧЕСКИ ВАЖНО — СДЕЛАТЬ ПРАВИЛЬНО)

Настраиваемый параметр: SPREAD_REDUCTION (по умолчанию 0.30) — общее снижение спреда в процентных пунктах (пп).
Это означает: если рыночный спред 2.8%, наши лимитные ордера создают спред 2.5% между собой (2.8 - 0.30 = 2.5).
Снижение делится поровну между обеими ногами: half_reduction = SPREAD_REDUCTION / 2.

Формулы:
  market_spread_pct = (sell_bid / buy_ask - 1) * 100
  half_reduction = SPREAD_REDUCTION / 2  # по умолчанию: 0.15
  adjusted_buy  = buy_ask  * (1 + half_reduction / 100)  # платим БОЛЬШЕ → цена покупки ВВЕРХ
  adjusted_sell = sell_bid * (1 - half_reduction / 100)  # получаем МЕНЬШЕ → цена продажи ВНИЗ

Проверка: (adjusted_sell / adjusted_buy - 1) * 100 ≈ market_spread_pct - SPREAD_REDUCTION

Пример: buy_ask=0.2338, sell_bid=0.2362, SPREAD_REDUCTION=0.30
  market_spread = 1.026%, half = 0.15
  adjusted_buy  = 0.2338 * 1.0015 = 0.23415
  adjusted_sell = 0.2362 * 0.9985 = 0.23585
  result_spread = (0.23585 / 0.23415 - 1) * 100 ≈ 0.726%  (1.026 - 0.30 = 0.726 ✔)

Сдвиг цены ПРОПОРЦИОНАЛЬНЫЙ (умножение на 1±дробь), НЕ абсолютное сложение.

# ПОТОК РАБОТЫ ПРИЛОЖЕНИЯ

## 1. Фаза настройки
- Спросить оператора position_size_usdt (одинаково для обеих ног, по умолчанию 100)
- Спросить оператора SPREAD_REDUCTION (по умолчанию 0.30 пп)
- Подключиться к Redis
- Инициализировать инстансы бирж (лениво — только когда придёт первый сигнал для этой биржи)

## 2. Цикл сигналов (состояние IDLE)
- Слушать Redis канал, блокировать до прихода сигнала
- Показать панель информации о сигнале: символ, биржи, цены, спред %, возраст
- Показать план входа: оригинальные цены, скорректированные цены, количества, спред до/после
- Спросить: «Входим в сделку? [Y/N]»
- Если N → пропустить, вернуться к прослушиванию

## 3. Открытие (если Y)
- Разместить ОБА лимитных ордера ПАРАЛЛЕЛЬНО (threading, не последовательно!)
  - Нога BUY: limit buy на спот-бирже по adjusted_buy
  - Нога SELL: limit sell на фьючерсной бирже по adjusted_sell (открывает шорт)
- Округлить количество и цену по точности биржи (market.precision.amount / .price)
  Обработать оба варианта: int precision (знаки после запятой) и float precision (размер шага)

## 4. Мониторинг заполнения
- Поллить статус ордеров каждую ~1 секунду
- Показывать live-таблицу: order_id, биржа, сторона, символ, запланированные цена/количество, заполненное количество, средняя цена, статус
- Использовать Rich Live для обновления терминала в реальном времени
- Как только ОБА ордера заполнены (status=closed) → перейти к монитору PnL
- Таймаут 5 мин: спросить оператора — отменить или ждать дальше

## 5. Монитор PnL
- Получать тикеры с обеих бирж каждые 1-2 секунды
- Рассчитывать и показывать:
  - PnL покупки (лонг спот): (текущий_bid - цена_входа) * заполненное_количество
  - PnL продажи (шорт фьючерс): (цена_входа - текущий_ask) * заполненное_количество
  - Общий PnL
  - Спред входа vs текущий спред
  - Длительность позиции
- Использовать Rich Live дисплей
- Оператор вводит 'close' или нажимает Ctrl+C для закрытия

## 6. Закрытие
- Разместить ОБА маркет-ордера ПАРАЛЛЕЛЬНО:
  - Закрытие ноги BUY: market SELL (продать спотовую позицию)
  - Закрытие ноги SELL: market BUY (закрыть фьючерсный шорт)
- Показать финальные результаты

## 7. Логирование
- Сохранить полный лог в logs/trade_{SYMBOL}_{YYYYMMDD_HHMMSS}.log
- Содержимое лога: сигнал, параметры, обе ноги (вход + закрытие), таймлайн, все события
- Показать лог в терминале
- Вернуться в IDLE → следующий сигнал

# КРИТИЧЕСКОЕ ПРАВИЛО БЛОКИРОВКИ
Пока идёт цикл сделки (от показа сигнала до лога), НЕ обрабатывать новые сигналы.
После завершения цикла взять следующий сигнал из потока Redis.
Проверить возраст сигнала — пропустить если старше SIGNAL_MAX_AGE_SEC (по умолчанию 30с).

# ОБРАБОТКА ОШИБОК
- Один ордер упал, другой успешен: показать ошибку, спросить оператора — отменить успешный или продолжить с одной ногой
- Оба ордера упали: показать ошибки, вернуться в IDLE
- Redis отключился: переподключение с экспоненциальным backoff
- Ордер не заполняется 5 мин: спросить оператора
- Ордер закрытия упал: показать ошибку, повторить попытку
- Непарсимый сигнал: пропустить, лог в stderr
- Ctrl+C во время открытой позиции: закрыть позицию, сохранить лог, выйти чисто

# ВАЖНЫЕ ЗАМЕЧАНИЯ
- Fees и slippage НЕ учитываются при расчёте размера позиции
- Фактические USDT на ногу могут немного отличаться из-за округления — это нормально
- Приложение однопоточное в главном цикле, только размещение ордеров использует потоки
- SPREAD_REDUCTION задаётся в процентных пунктах (пп), НЕ как множитель
- Все цены показывать оператору в читаемом виде (спред как '2.65%', а не '1.0265')
